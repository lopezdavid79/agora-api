Quick start

  1. Clonar e instalar: npm install
  2. Configurar .env con DB_HOST, DB_USER, DB_PASS, DB_NAME, JWT_SECRET
  3. Iniciar: npm start o npm run dev (nodemon)

El servidor arranca en http://localhost:3000 y crea las tablas automáticamente si no existen.

POST /api/auth/login

Body:

{
  "email": "candidato@ejemplo.com",
  "password": "miPassword123"
}

Respuesta 200 (exitosa):

{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "usuario": {
    "id": 1,
    "email": "candidato@ejemplo.com",
    "rol": "Candidato",
    "nombre": "Sandra",
    "apellido": "Polutranka",
    "estadoPerfil": "Pendiente"
  }
}

Errores posibles:

  • 400 — validación (email inválido o password vacío)
  • 401 — credenciales incorrectas
  • 403 — cuenta deshabilitada

Nota: El token JWT incluye id, email y rol. Se envía como Authorization: Bearer <token>.

GET /api/auth/me

Requiere token. Devuelve el usuario completo (sin password) con su perfil asociado.

PUT /api/auth/cambiar-password

Requiere token.

Body:

{
  "passwordActual": "vieja123",
  "passwordNueva": "NuevaSegura456"
}

Requisitos de la nueva contraseña:

  • Mínimo 8 caracteres
  • Al menos una mayúscula
  • Al menos un número

Autenticación y autorización

Flujo de login

  1. El cliente envía email + password a POST /api/auth/login
  2. Express-validator valida que email sea válido y password no esté vacío
  3. Se busca el usuario por email en la base de datos (incluyendo su perfil)
  4. Si no existe → respuesta 401 "Credenciales incorrectas"
  5. Si el usuario no está activo → respuesta 403 "Cuenta deshabilitada"
  6. Se compara la contraseña con bcrypt
  7. Si no coincide → respuesta 401 "Credenciales incorrectas"
  8. Se genera un JWT con { id, email, rol } firmado con JWT_SECRET
  9. Se actualiza ultimoAcceso (de forma asíncrona, no bloquea la respuesta)
  10. Se devuelve { token, usuario } con los datos del candidato

Middlewares de autenticación

MiddlewareFunción
verificarTokenExtrae el token del header Authorization: Bearer <token>, lo verifica y setea req.usuario
soloRoles(...roles)Permite el acceso solo si req.usuario.rol está en la lista de roles permitidos

Notas técnicas

  • Las tablas se crean automáticamente al iniciar el servidor con sequelize.sync({ force: false }). No borra datos existentes.
  • Soft delete en documentos (columna deleted_at), aunque no está implementado en las queries actuales.
  • Los campos en la DB usan snake_case; en el código se accede en camelCase gracias a la configuración underscored: true de Sequelize.
  • CORS configurado para aceptar peticiones del frontend en http://localhost:5173 (puerto default de Vite).
  • El password "AgoraArgentina2026!" funciona para la mayoría de los candidatos de prueba.
